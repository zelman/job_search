The rescorer is roughly **65% aligned** with v2.17 by item count. Divergences cluster in two areas: **(a) Config table threshold values are tighter than v2.17 in some places, looser in others**, and **(b) the Score-Apply/Watch/Pass bucket math is decoupled from v2.17's Strong-Fit/Good-Fit/Marginal/Skip framework** — different labels, different thresholds, no translation layer.

**Most consequential divergence:** The rescorer's `EMPLOYEE_HARD_CAP` is **1000** in the live Config table, but v2.17's hard kill is **350**. Records with 350-1000 employees pass the rescorer's gate but fail the lens spec's gate — the entire 350-1000 band is a silent over-permissive hole.

**Most consequential gap:** Several v2.17 bonuses (Series A +50, Series B +30, 0-50 emp +50, Builder language +40, Leadership title +15, Healthcare +15, VC-backed +10, Founded <3y +10) are not deterministically applied. They're delegated to the Claude eval call via the Build Eval prompt, which gives Claude latitude rather than enforcing v2.17's point arithmetic. The rescorer's deterministic bonuses are limited to: `REBUILD_BONUS +20`, `BUILDER_PHRASE_BONUS +15` (lens says +40), and `domain_distance_modifier` ranges (Claude-decided).

**Cross-reference with today's Phase 1 work (commit 71fed4a):** The PE-list safety patch only touched the Enrich pipeline's hardcoded `peFirms` array. The rescorer is already config-driven and reads PE Firms from Airtable `tblU2izcb0wnCNMuV`. So 71fed4a does NOT propagate to the rescorer. **Worse:** the Airtable PE Firms table still contains Insight Partners, NEA, Battery, Thrive, and Marlin — exactly the mixed-arm firms the user explicitly removed from the Enrich pipeline today. The rescorer therefore uses a more aggressive PE-DQ list than the Enrich pipeline does, post-71fed4a.

## Gaps Requiring Action (sorted by implementation complexity)

### Tier 1: Config Fetcher value tweaks (smallest)

These are 1-row Airtable edits in the Config table. No code change needed.

1. **`EMPLOYEE_HARD_CAP`: 1000 → 350.** Lens spec says hard kill at 350. Current 1000 silently lets the entire 350-1000 band pass. Highest-impact edit.
2. **`FUNDING_HARD_CAP`: $200M → $500M.** Current is *tighter* than lens (kills viable companies in $200M-$500M band). Lens specifies $500M as the kill threshold and $200M-$500M as a penalty zone instead.
3. **`SCORE_APPLY_THRESHOLD`: 70 → 80** (matching `strong_fit: 80` in lens) **OR** add a separate `SCORE_GOOD_FIT_THRESHOLD: 60` for the Good Fit tier.
4. **`SCORE_WATCH_THRESHOLD`: 25 → 40** (matching `marginal: 40` in lens). Current 25 is far below v2.17's marginal floor.
5. **`COMPANY_AGE_HARD_CAP`: 8 → remove** (or pair with employee_count check). Lens stalled_company is `founded >10y AND <200 emp` — a single age cap kills any 9-year-old company regardless of size.
6. **`BUILDER_PHRASE_BONUS`: 15 → 40.** Lens says builder language is the highest-weighted positive signal at +40.
7. **`EMPLOYEE_SOFT_CAP`: 200 → 150.** Rebuild exception range should start at 150 per v2.17.

### Tier 2: Build Eval prompt edits (small)

JS-string edits inside one node. No data flow change.

8. **Add VERY LOW CS Hire Readiness tier** ("Large mature CS org 20+ with established leadership hierarchy"). Currently 3 tiers; v2.17 has 4.
9. **Add Healthcare B2B SaaS as +15 flat bonus** (vs current `domain_distance_modifier` +3 to +5). Or keep modifier and document that v2.17's +15 is intentionally not enforced.
10. **Replace `domain_distance_modifier` with explicit DQ for v2.17 domain_expertise list** (Pharma Marketing, Healthcare Agency, FinServ, Legal, AdTech, Gov, Web3, Biotech, HR Tech). Current implementation lets Claude decide; lens says hard DQ.
11. **Add Series A/B/C stage bonuses (+50/+30/+10)** to Build Eval prompt as deterministic point ranges, not Claude-decided.
12. **Add 0-50 emp +50 bonus, VC-backed +10 bonus, Founded <3y +10 bonus** to Build Eval prompt.

### Tier 3: New field passthroughs through Build Persona / Parse Enrich (medium)

13. **Pre-Brave `business_model` classification.** Currently Parse Enrich detects Hardware/CX-Tooling but not the full v2.17 list (B2C, Cleantech, Robotics, AgTech, Aquaculture). Either expand Parse Enrich detection or pass description text through to Build Eval for Claude classification with the v2.17 list.
14. **Customer Persona enterprise exception logic.** Current rule: `developer AND emp <50 = auto-pass`. v2.17 rule: `developer AND <50 enterprise signals = auto-pass`, with exception for "50+ emp AND 2+ enterprise signals (Fortune 500, sales team, SSO/SAML, on-prem, ACV/ARR mentions, procurement)". Need enterprise-signal detection in Parse Enrich, then logic update in Parse Persona.
15. **Network bonus deterministic scoring.** `hasNetworkConnection` is populated by Match Jobs & Network but not converted to lens's +15/+10 founder/warm-intro bonuses. Add to Parse Eval after Claude returns.
16. **Data Validation Flags** (Seed >$20M, Late stage <10 emp, Series D <$20M). Add to Parse Enrich as warnings, surface in `flags_for_review`.

### Tier 4: New node or logic branch (large)

17. **Decision threshold framework rewrite.** Rescorer's APPLY/WATCH/PASS bucket math doesn't translate to lens's STRONG FIT/GOOD FIT/MARGINAL/SKIP. Either rename buckets and align thresholds OR add a translation layer in Parse Eval that computes both. Today, Status field gets set by APPLY/WATCH/PASS terminology while score range disagrees with v2.17 thresholds. This is the most user-facing divergence.
18. **PE Firms Airtable cleanup matching today's Enrich-side decisions.** `tblU2izcb0wnCNMuV` still contains Insight, NEA, Battery, Thrive, Marlin. Either remove them (so rescorer matches Enrich-side commit 71fed4a) or implement acquisition-recency logic so growth-stage portfolios from these firms aren't blanket-DQ'd. Same architectural decision as the Enrich pipeline; both sources need to agree.
19. **Bucket recalculation bug.** Parse Eval only recomputes `bucket` when `scoreCapped` or `funding_staleness_modifier < 0`. This means the +20 rebuild bonus or the v5.0 hardcoded headcount penalties can change `finalScore` without changing `bucket` — record gets a Status that doesn't match its score. Either always recalculate bucket OR document that Claude's bucket is authoritative regardless of post-hoc adjustments.
