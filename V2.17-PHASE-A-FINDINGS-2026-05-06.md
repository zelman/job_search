## Q1 — Does the rescorer guarantee Config Fetcher completes before Parse Enrich?

**Answer:** **Yes. The connection graph enforces it via an `n8n-nodes-base.merge` node in `combineAll` mode.**

**Evidence:**
- File: `Funding Alerts Rescore v5.0-standalone.json`, `connections` block.
- Edge map:
  - `Capture` → `Config Fetcher` (input 0)
  - `Capture` → `Brave` (input 0)
  - `Config Fetcher` → `Merge Config` (input 0)
  - `Brave` → `Merge Config` (input 1)
  - `Merge Config` → `Parse Enrich` (input 0)
- `Merge Config` node parameters: `{"mode": "combine", "combineBy": "combineAll"}`. n8n's `combineAll` merge waits for ALL inputs before emitting downstream — by definition Config Fetcher must complete before Parse Enrich runs.
- Parse Enrich reads config via `$('Config Fetcher').first().json` (n8n cross-node accessor). The accessor returns the cached output of the named node from the current execution context; the upstream merge guarantees that cache is populated.

**Fallback behavior:** Parse Enrich has `const EMPLOYEE_HARD_CAP = config.EMPLOYEE_HARD_CAP || 150;` style fallbacks at the top of its jsCode (~15 such constants). These only fire if the Airtable Config GET response itself is empty/malformed (data integrity issue, not a race). Notable: the fallback values disagree with the live Config table (e.g., fallback `EMPLOYEE_HARD_CAP=150` vs live Config `1000`) — if Config Fetcher ever silently failed, the rescorer would suddenly become *more* restrictive than current production behavior. Worth a fault-tolerance test in a separate session.

**Implication for Phase B:** Editing the Airtable Config table is sufficient to change rescorer behavior — no rescorer node modification required.

## Q2 — Series C handling: live code path

**Answer:** **Series C is treated as PENALTY/MANUAL REVIEW, not auto-DQ and not auto-PASS. This MATCHES v2.17 spec.** The contradictory comments in Parse Enrich are stale version markers, not live behavior.

**Evidence:**
- File: `Funding Alerts Rescore v5.0-standalone.json`, Parse Enrich node, `jsCode`.
- Stale comment block at top of Parse Enrich:
  ```
  // v4.15: Stage Gate + Mature Company Detection (Mar 2026)
  // - Series C or later = auto-PASS
  // - Stage unknown but employees > 500 OR funding > $200M OR valuation > $500M = auto-PASS
  // - Founded > 12 years ago = auto-PASS
  ```
- Live code (after `// v5.0 fix:` comment):
  ```javascript
  // v5.0 fix: Series C with <300 employees goes to WATCH, not auto-DQ
  if (isLateStageByPattern) {
    const emp = employeeCount || 0;
    if (fundingStage === 'Series C' && emp < 300) {
      flagsForReview.push('Series C with ' + emp + ' employees — manual review');
    }
    // ...else fallthrough to other stage checks
  }
  ```
- Series D and Series E are the only stage-based hard DQs:
  ```javascript
  else if (['Series D', 'Series E'].includes(fundingStage)) 
    disqualifiers.push('Late stage (' + fundingStage + ')');
  ```
- Series C with employees ≥ 300 may still get killed by the employee cap (not by Series C check itself).

**Trace of a Series C / 200-employee record through the rescorer:**
1. `fundingStage = 'Series C'` set from regex on Brave text or Airtable `Stage` field.
2. `isLateStageByPattern = true` (Series C matches LATE_STAGE_PATTERNS).
3. Series C branch fires: `flagsForReview.push('Series C with 200 employees — manual review')`.
4. Hard gates (PE, Public, headcount > 1000, funding > $200M, etc.) checked — none hit.
5. `disqualifiers` array stays empty for this record.
6. `isAutoDisqualified = false`.
7. Record proceeds to persona classification → eval → bucket assignment.
8. Final `Status` determined by Claude eval + post-hoc adjustments — not by Series C check.

**Note on lens spec divergence:** v2.17 says `Series C: -5 pts` (deterministic penalty applied at scoring time). The rescorer's `flagsForReview` is informational only — there's no -5 point penalty arithmetic in Parse Eval that responds to Series C presence. So Series C is not auto-DQ (matches v2.17), but the -5 penalty is also not enforced (diverges from v2.17 — Claude is free to score Series C records however it wants based on the prompt). **Listed as a Tier 2 gap in the audit; not investigated further this session.**

## Q3 — Watch threshold conflict: which value wins?

**Answer:** **The hardcoded threshold (40) in Parse Eval wins WHEN bucket recalculation triggers. Otherwise Claude's bucket passes through unchanged. The Config table value `SCORE_WATCH_THRESHOLD=25` is dead code — loaded but never referenced.**

**Evidence:**
- File: `Funding Alerts Rescore v5.0-standalone.json`, Parse Enrich node, jsCode top section:
  ```javascript
  const SCORE_WATCH_THRESHOLD = config.SCORE_WATCH_THRESHOLD || 40;
  const SCORE_APPLY_THRESHOLD = config.SCORE_APPLY_THRESHOLD || 70;
  ```
  These constants are declared. Grep across the entire Parse Enrich jsCode finds **zero references** to either constant after declaration. They are dead variables.
- File: same JSON, Parse Eval node, jsCode bucket recalc block:
  ```javascript
  // v4.6: Recalculate bucket based on capped score
  let bucket = evaluation.bucket;  // From Claude
  if (scoreCapped || prev.funding_staleness_modifier < 0) {
    if (finalScore >= 70) bucket = 'APPLY';      // hardcoded 70
    else if (finalScore >= 40) bucket = 'WATCH'; // hardcoded 40
    else bucket = 'PASS';
  }
  ```
  No Config plumbing. The thresholds 70 and 40 are literal in this file.
- The two nodes do not share variable scope. Parse Enrich's loaded Config values do not propagate to Parse Eval — Parse Eval only receives `prev` (from Build Eval) and `$json` (Claude API response).

**When does recalculation fire?**
- Only if `scoreCapped == true` (CS hire likelihood is `unlikely` or `low`, score capped at 65 or 75 respectively)
- OR if `prev.funding_staleness_modifier < 0` (early-stage company with 3+ year stale funding)
- In all other cases, `bucket` retains whatever value Claude returned in the API response.

**Implication:** Even if the user updates Config table `SCORE_WATCH_THRESHOLD` and `SCORE_APPLY_THRESHOLD`, those values won't change rescorer behavior. The bucket math is fully decoupled from Config. To bring threshold tuning into v2.17 alignment, the Parse Eval node code itself needs editing (Tier 2 work), or Config plumbing needs adding from Parse Enrich → Build Eval → Parse Eval (Tier 3 work). **Important context for the next planning session.**

## Q5 — Job posting field consumption in Build Eval prompt

**Answer:** **Case (b) — passed but not weighted. The fields are populated by Match Jobs & Network and written to Airtable by Parse Eval, but the `evalPrompt` string sent to Claude does not reference them. Claude scores CS readiness blind to these signals.**

**Evidence:**
- File: `Funding Alerts Rescore v5.0-standalone.json`, Build Eval node, jsCode.
- String search for field names in the entire Build Eval node:
  - `hasActiveJobPosting` — 2 occurrences, both in the **output JSON object** (`return { json: { ... hasActiveJobPosting: c.hasActiveJobPosting, ... } }`). Zero occurrences in the `evalPrompt` string.
  - `hasCXJobPosting` — same pattern, 2 output-only.
  - `matchingJobTitles` — same pattern, 2 output-only.
  - `job_posting`, `Job Posting`, `job opening`, `CX job` — 0 occurrences in the prompt string.
- Confirmed by isolating the `evalPrompt = '...';` assignment and running a substring search over its contents: no job-posting field reference of any kind.

**Data flow:**
1. Match Jobs & Network node populates `hasActiveJobPosting`, `hasCXJobPosting`, `matchingJobTitles` from Airtable Job Listings table cross-reference.
2. Build Persona node passes these through unchanged.
3. Build Eval node passes these through to its output JSON (so they survive to Parse Eval) but does NOT include them in the prompt sent to Claude.
4. Parse Eval writes them to Airtable as record fields.
5. Claude never sees them, so they cannot influence the score.

**What Claude actually sees** (from the Build Eval prompt):
- `company_name`, `company_url`, `description`, `funding_stage`, `employee_count`, `founded_year`, `customer_persona`
- Aging flag note (if applicable)
- Context flags: CX TOOLING, STALE FUNDING, SUSPICIOUS EMPLOYEE COUNT, REBUILD SIGNAL
- Scoring rubric (CS Hire Readiness HIGH/MEDIUM/LOW tiers, sector_match, role_type, etc.)

**Implication for the leadership-gap heuristic rebuild proposed in `PHASE1-ENRICHMENT-BUG-2026-05-05.md`:** This is a **small Tier 2 fix**, not a plumbing rebuild. The data is already collected, already passing through to Parse Eval, just not included in the prompt. Adding lines like `${c.hasCXJobPosting ? 'HAS CX JOB POSTING' : 'NO CX JOB POSTING'}` and `Matching titles: ${c.matchingJobTitles}` to `evalPrompt` would let Claude condition CS readiness scoring on these signals. Estimated implementation: 5-10 minutes of Build Eval prompt editing, no schema work, no new node, no field passthrough work.

## Phase B — Credential Rotation + Validation (2026-05-06 follow-up)

### Why this section exists

Yesterday's Phase B blocked at the rescorer execution: every cycle failed with `Authorization failed - please check your credentials` on the `Update DQ (HTTP)` node. The fix took multiple iterations — surfacing useful debugging signal each time — and the validation test eventually completed.

### Credential rotation diagnosis (3 distinct failure modes)

| # | Action | Resulting failure | Lesson |
|---|--------|-------------------|--------|
| 1 | User minted new PAT, replaced token value in n8n UI | `Authorization failed - please check your credentials` | Token rotation alone insufficient |
| 2 | User added `Bearer ` prefix to the value | `Credential with ID "LCYvQYofV9XXnPrk" does not exist for type "httpHeaderAuth"` | Bearer prefix was missing originally — but irrelevant; new error revealed the root cause was deeper |
| 3 | User re-rotated with another new token | Same: `Credential with ID "LCYvQYofV9XXnPrk" does not exist` | Updates to a new credential don't help if the workflow nodes reference an old, deleted credential ID |
| 4 | User re-linked the 3 HTTP write nodes to the working credential in n8n UI (Path A from session) | `success` (10.5s) — Skild AI auto-DQ'd, write succeeded | Re-link is the actual fix |

**Root cause:** the original `Airtable Token` httpHeaderAuth credential (ID `LCYvQYofV9XXnPrk`) was at some point deleted or recreated, breaking node references. The token value itself was probably fine — the credential entity referenced by the nodes simply didn't exist anymore.

**Operational lesson:** in n8n, **update credential values in place, do not delete and recreate** — recreation generates a new credential ID and orphans every node reference. If recreation is unavoidable, every consuming node must be re-linked.
